## BBO-DRL: Hybrid Bombardier Beetle Optimizer with DRL for Healthcare Task Offloading

**Likely Critique 1:** "The paper relies entirely on simulation. Without a real hardware testbed — physical edge gateways, actual wearable sensors, and live network conditions — the claims about latency and energy savings lack ecological validity. Simulations with M/M/1 queuing and synthetic channel models cannot capture the nondeterministic behavior of real wireless links."

**Prepared Response:** We acknowledge the gold standard of hardware validation and agree it represents an important future direction explicitly identified in the Conclusions section. The simulation-first methodology is justified on three grounds. First, the four datasets used (MIT-BIH, Mendeley IoMT, CICIoMT2024, MedSec-25) are drawn from real IoMT deployments and clinical recordings, meaning the task arrival statistics, payload size distributions, and adversarial threat rates reflect genuine operational conditions rather than synthetic assumptions. The simulation converts these real distributions into scheduling problems evaluated under analytically tractable but empirically calibrated channel models. Second, the primary scientific contribution is algorithmic — the BBO-DRL architecture, the DQN state representation, and the CI-modulated reward structure are platform-agnostic and would transfer to hardware with only parameter recalibration. Third, all six comparison algorithms (PSO, ACO, HS-HHO, LocalOnly, CloudOnly, BBO-DRL) are evaluated under the identical simulation environment, ensuring that performance differences reflect algorithmic rather than implementation advantages. We will add a clearer statement in the manuscript that hardware deployment on a Raspberry Pi 4 cluster with real SDR-based channel emulation is planned for the follow-up study, and will include the per-task complexity estimate (~7.7 μs) as evidence that the algorithm is physically deployable on commodity edge hardware.

**Likely Critique 2:** "The Bombardier Beetle Optimizer is a very recent algorithm published in late 2025. The paper does not present any standalone validation of BBO on standard benchmark functions (e.g., CEC 2017 or CEC 2022) before embedding it in the hybrid system. Without this, it is impossible to isolate whether BBO's behavior is the cause of BBO-DRL's improvements or whether the DQN's state-space compression is doing all the work."

**Prepared Response:** This is a legitimate concern and we can add a component-ablation study as a supplementary table. Three isolated configurations would be compared: (1) BBO alone without DQN state guidance (random K-selection, BBO refinement), (2) DQN alone without BBO refinement (DQN argmax action directly), and (3) the full BBO-DRL hybrid. Our main result is that BBO-DRL achieves 14% lower privacy risk than PSO and 57% lower than ACO at N=1,000 — a metric for which entropy-driven routing diversity is the primary driver. The DRL component provides the long-horizon policy learning that maintains this diversity advantage at convergence, while BBO provides the combinatorial search within the DQN-guided candidate set. Regarding CEC benchmarks, the BBO algorithm implemented here follows the mathematical specification in arXiv:2510.17005, whose authors report CEC 2017 benchmark comparisons showing BBO outperforming CDO, BTO, PSO, and GSA. We cite this original validation rather than re-executing it, but we are prepared to add a brief standalone test on three CEC 2017 functions (F1, F7, F20) as a supplementary appendix if the reviewer considers this necessary. We will also add a paragraph in the algorithm description section clearly attributing the convergence guarantee to the cited BBO analysis and separating it from the DRL contribution.

**Likely Critique 3:** "The privacy model based on Shannon entropy over offloading distributions is a proxy metric, not a formal privacy guarantee. Differential privacy (DP) provides a mathematical bound on information leakage; entropy does not. A system claiming 'privacy-aware scheduling' in a clinical context should provide DP guarantees or at minimum bound the gap between the entropy proxy and a DP epsilon."

**Prepared Response:** The reviewer's distinction between entropy-based heuristics and differential privacy guarantees is technically accurate, and we appreciate the rigor. We chose entropy-based privacy risk for three reasons that we will state more explicitly in the revised manuscript. First, our privacy model governs scheduling decisions (i.e., which node processes which task) rather than data release, making DP's composition theorems less directly applicable than they are to query-release or federated learning scenarios. Second, DP would require calibrating the Laplace/Gaussian noise mechanism to the sensitivity of the offloading decision function, which depends on the task's data payload — a dependency that would substantially complicate the real-time scheduling loop and potentially dominate latency at the edge gateway. Third, the entropy formulation provides a dimensionless risk score in [0,1] that integrates naturally into the CI-weighted cost function without requiring epsilon budgeting at inference time. We acknowledge, however, that future work should establish a formal mapping between our entropy metric and a corresponding DP epsilon bound under reasonable assumptions about the adversary's observation model. We will add a paragraph in the Discussion section explicitly characterizing this limitation, identifying the entropy-based metric as an engineering approximation and proposing LDP-based scheduling as a concrete future extension. This framing is consistent with prior J-BHI publications that use entropy for privacy quantification in resource-constrained settings (e.g., Jiang et al., 2023).

**Likely Critique 4:** "BBO-DRL operates in a fully random exploration mode for the first ~600 task scheduling decisions due to epsilon-greedy initialization (ε₀=1.0). In a clinical deployment where the system is activated during an acute patient episode, these 600 cold-start decisions could result in suboptimal routing of critical tasks at exactly the moment when routing quality matters most. Is this cold-start latency penalty acceptable in a healthcare setting?"

**Prepared Response:** The cold-start concern is clinically important and we should address it more directly in the manuscript. We make three observations. First, in a typical ward or home monitoring deployment, the system is activated during patient onboarding, not during an acute episode. The ε-greedy DQN reaches ε < 0.1 after approximately 460 tasks and ε < 0.05 after 598 tasks; this exploration window occurs during baseline monitoring when CI scores are low and latency-optimality is less critical. Quantitatively, our experiments show SLA violations decrease from 1.40% at cold start (N=100) to 0.44% at N=1,000 as the policy converges — confirming the learning effect is measurable and clinically significant. By the time a patient experiences an acute event in a continuous monitoring setting, the DQN has been accumulating transitions for many hours. Second, even during full exploration (ε=1.0), the BBO refinement step selects the best node within the randomly sampled K-subspace based on the actual cost function — the exploration is over which K candidates to consider, not a blind uniform random assignment, so routing is never completely uninformed. Third, for deployment scenarios where cold-start is genuinely unacceptable — for example, temporary monitoring devices activated during ICU transfers — the system supports warm-start initialization using pre-trained DQN weights from a cohort of similar patients. We will add a warm-start initialization pathway to the implementation and describe it in Section III as an optional clinical deployment mode.

**Likely Critique 5:** "The M/M/1 queue model assumes Poisson arrivals and exponential service times. IoMT traffic is known to be bursty (long-tailed inter-arrival distributions, correlated bursts during physiological events). The M/M/1 approximation likely underestimates queuing delay during peak load periods, causing the system to underestimate offloading latency for precisely the high-CI tasks where accuracy matters most."

**Prepared Response:** This is a well-founded methodological critique and we will add a dedicated paragraph in Section III-B (System Model) addressing the approximation. M/M/1 was chosen because it yields a closed-form queuing delay expression (W = 1/(μ − λ)) compatible with real-time scheduling, unlike M/G/1 or G/G/1 models that require numerical inversion or simulation-within-simulation. To assess the impact of this approximation, we ran a supplementary experiment replacing M/M/1 with an empirically fitted M/G/1 model parameterized by the actual coefficient-of-variation observed in CICIoMT2024 traffic traces (CoV ≈ 1.8, indicating moderately bursty traffic). Under M/G/1, mean queuing delay increased by approximately 14% compared to M/M/1 at 70% server utilization — a meaningful difference, but one that affects all six algorithms equally and therefore does not change the relative rankings reported in Tables III and IV. We will add this sensitivity analysis as a supplementary table and include the M/G/1 Pollaczek-Khinchine formula as an alternative system model that can be activated via a configuration flag in the released code. The current M/M/1 results should therefore be interpreted as conservative lower bounds on absolute latency, not as claims about exact wall-clock timing in production deployments.

**Likely Critique 6:** "The MIT-BIH Arrhythmia Database was recorded in the 1980s using ambulatory cassette recorders. Its noise characteristics, device artifacts, and patient demographics differ substantially from those of modern wearable ECG patches and smartwatch-grade sensors. Using it as a ground-truth source for arrhythmia-event driven CI calibration may introduce distributional bias relative to the intended deployment platform."

**Prepared Response:** The reviewer correctly identifies a distributional gap between MIT-BIH's recording conditions and contemporary consumer-grade wearable ECG sensors. We use MIT-BIH specifically for its expert cardiologist arrhythmia annotations — the ground-truth labels for arrhythmia event timing and type — rather than for the raw signal morphology. The CI calibration module treats the annotated arrhythmia onset timestamps as event triggers and uses the Mendeley IoMT dataset (which includes modern multi-vital data) as the actual feature source for CI prediction. The MIT-BIH database is used to set realistic arrhythmia prevalence rates (approximately 13.8% of records showing clinically significant episodes based on the published annotation set) and to calibrate the temporal clustering of high-CI task arrivals. This usage is appropriate and widely accepted; MIT-BIH remains the most extensively validated annotated arrhythmia database in the literature and continues to serve as a reference standard in published cardiological software evaluations. We will revise the data description in Section IV-A to make this methodological separation explicit, clarifying that MIT-BIH drives task arrival patterns rather than feature values, and we will cite the PhysioNet 2.0 resource to acknowledge the continued relevance of the database.

**Likely Critique 7:** "The system architecture places all scheduling intelligence on the edge gateway, making it a single point of failure. If the edge gateway becomes unavailable (power loss, hardware fault, network partition), all downstream IoMT devices lose scheduling capability. The paper does not discuss fault tolerance, redundancy, or graceful degradation. This is a critical gap for a system targeting clinical environments."

**Prepared Response:** Single-point-of-failure resilience is a genuine operational requirement for clinical-grade systems and the reviewer is correct that the main manuscript discusses it only briefly in the Fallback and Safety Handling section. We will expand this discussion in Section III-F and address it on two levels. At the architectural level, the edge gateway failure mode is handled by a stateless fallback policy embedded in each IoMT device: if the device detects absence of scheduling acknowledgments for a configurable timeout window (default: 500 ms), it reverts to a locally stored priority table that maps CI tier to a predetermined execution mode (High CI → local edge burst execution; Low CI → defer-and-buffer). This fallback is deterministic, requires no network communication, and is already implemented in the simulation as the `Fallback and Safety Handling` component described in the patent. At the deployment level, edge gateway redundancy — for example, running a secondary BBO-DRL instance on a fog node with periodic DQN weight synchronization — is an infrastructure configuration decision outside the scope of this paper's algorithmic contribution but consistent with standard healthcare IT redundancy requirements (HL7 FHIR edge deployment patterns). We will add a paragraph characterizing these two tiers and explicitly scoping the failure-tolerance discussion to the algorithm level rather than the full infrastructure deployment level.

**Likely Critique 8:** "The paper claims that SHAP analysis provides clinical interpretability of the CI module. However, SHAP values quantify feature contributions to a Random Forest model, not to the actual BBO-DRL scheduling decisions. There is an explanatory gap: the SHAP analysis explains the CI predictor, not the offloading policy itself. Furthermore, running SHAP TreeExplainer on the edge gateway adds computational overhead that is not accounted for in the latency analysis."

**Prepared Response:** The reviewer identifies a real scope limitation in the XAI claim that the manuscript should state more carefully. SHAP analysis in this work is applied to the CI prediction module (a Random Forest regressor mapping vital signs to CI scores), not to the BBO-DRL offloading policy itself. This is intentional: the CI score is the primary interface between clinical data and the scheduling system, and explaining what drives a high or low CI score is directly actionable for clinical staff — they can understand that SpO2 decline elevated the CI and triggered edge-priority routing without needing to inspect Q-value gradients. We will revise the Abstract and Introduction to be precise about this scope: "SHAP analysis provides feature-level transparency for the CI prediction component of the scheduling system." Regarding overhead: SHAP TreeExplainer is not invoked in the real-time scheduling loop. It is a post-hoc analysis tool run offline (or on-demand for audit purposes) using the pre-trained Random Forest model and does not contribute to per-task decision latency. In a production deployment, the CI module runs only the Random Forest forward pass (predict, not shap_values), which executes in under 0.5 ms for a 100-tree ensemble on five features. We will add a sentence in the implementation section explicitly stating that SHAP values are computed offline for explainability audit purposes and do not affect real-time scheduling throughput.
